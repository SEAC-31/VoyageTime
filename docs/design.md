# VoyageTime Sprint 03 Design Notes

## Persistence architecture

VoyageTime uses Room as a local SQLite abstraction. The app follows the Repository pattern: UI screens call ViewModels, ViewModels call domain repositories, and repository implementations access Room DAOs or Firebase Authentication.

Room components:

- `VoyageTimeDatabase`: main Room database.
- Entities: `TripEntity`, `ItineraryItemEntity`, `UserEntity`, `AccessLogEntity`.
- DAOs: `TripDao`, `ItineraryItemDao`, `UserDao`, `AccessLogDao`.
- Repositories: `TripRepositoryImpl`, `ItineraryRepositoryImpl`, `UserRepositoryImpl`, `FirebaseAuthRepositoryImpl`.

## Database schema

### trips

Stores trips created by users.

Important columns:

- `id`: local autogenerated primary key.
- `user_id`: Firebase UID of the owner.
- `destination`: trip destination.
- `country`: trip country.
- `start_datetime`: start date/time.
- `end_datetime`: end date/time.
- `duration_days`: integer duration.
- `budget_amount`: integer budget.
- `status_label`: trip state label.
- `image_res`: default drawable resource.
- `cover_image_uri`: optional image selected from Gallery.
- `created_at`: local creation datetime.

Trips are filtered by `user_id`, so each logged-in user only sees their own trips.

Validation:

- Empty destination/country is rejected in UI.
- Past dates are rejected.
- End date before start date is rejected.
- Duplicate destination names for the same user are rejected through `TripDao.isTripDestinationTakenForUser(...)`.

### itinerary_items

Stores itinerary events for a trip.

Important columns:

- `id`: local autogenerated primary key.
- `trip_id`: related trip.
- `day_number`: itinerary day.
- `section`: morning, afternoon, evening or notes.
- `title`: event title.
- `location`: event location.
- `cost_amount`: integer cost.
- `scheduled_at`: event datetime.
- `notes`: optional notes.
- `created_at`: local creation datetime.

### users

Stores local user profile information required by T4.1.

Important columns:

- `firebase_uid`: Firebase UID and primary key.
- `username`: local username.
- `email`: login email.
- `birthdate`: date field.
- `address`: user address.
- `country`: user country.
- `phone`: user phone.
- `accept_emails`: whether the user accepts receiving emails.
- `created_at`: local creation datetime.

Validation:

- Username is required.
- Username must be unique. This is checked with `UserDao.isUsernameTaken(...)` before insert/update.

### access_log

Stores login/logout operations required by T4.4.

Important columns:

- `id`: local autogenerated primary key.
- `user_id`: Firebase UID.
- `event_type`: `LOGIN` or `LOGOUT`.
- `timestamp`: datetime of the access event.

## Authentication

Firebase Authentication is used for email/password authentication.

Registration flow:

1. User completes the registration form.
2. Firebase user is created.
3. Local user profile is inserted in Room.
4. Verification email is sent.
5. User is logged out until email verification is completed.

Login flow:

1. Firebase signs in with email/password.
2. Firebase user is reloaded.
3. If `isEmailVerified` is false, login is blocked and user is signed out.
4. If verified, the user enters the app.
5. A `LOGIN` event is written to `access_log`.

Logout flow:

1. A `LOGOUT` event is written to `access_log`.
2. Firebase signs out.
3. The app returns to Login.

## Logcat tags

Authentication and database operations can be followed using these tags:

- `FirebaseAuthRepository`: login, register, email verification, password reset, logout.
- `LoginViewModel`: login validation and login UI errors.
- `RegisterViewModel`: registration validation and local/Firebase errors.
- `ForgotPasswordViewModel`: recovery validation and Firebase errors.
- `TripRepository`: trip insert/update/delete, duplicate trip validation and DB errors.
- `ItineraryRepository`: itinerary insert/update/delete and errors.
- `UserRepository`: local user insert/update, username uniqueness and access logs.
- `CreateTripViewModel`: create trip validation and database errors.

#### Email verification rule

The app requires Firebase email verification before allowing the user to enter the main application.

The login flow is:

1. The user enters email and password.
2. Firebase executes `signInWithEmailAndPassword(...)`.
3. The authenticated Firebase user is reloaded with `user.reload()`.
4. If `user.isEmailVerified == true`, the user is allowed into the app.
5. If `user.isEmailVerified == false`, the app calls `signOut()` and shows an error message.

This prevents non-verified users from accessing trips and local user data.

### T5.2 - Data validation

The application validates trip data before inserting or updating records in the Room database.

#### Validations implemented

| Validation | Location | Expected behavior |
|---|---|---|
| Empty destination | `CreateTripViewModel` / Trips edit form | The button to create trip stays disabled. |
| Empty country | `CreateTripViewModel` / Trips edit form | The button to create trip stays disabled. |
| Invalid date range | `CreateTripViewModel` / Trips edit form | Impossible to select in the DatePicker. |
| Start date in the past | `CreateTripViewModel` / Trips edit form | Impossible to select in the DatePicker. |
| End date before start date | `CreateTripViewModel` / Trips edit form |Impossible to select in the DatePicker. |
| Duplicate trip destination for the same user | `TripRepositoryImpl` | The trip is rejected before insertion. |

#### Duplicate trip validation

Duplicate trip names are checked per logged user. This means two different users may have a trip with the same destination, but the same user cannot create two trips with the same destination.

Example:

| User | Destination | Result |
|---|---|---|
| userA@test.com | Paris | Allowed |
| userA@test.com | Paris | Rejected |
| userB@test.com | Paris | Allowed |
